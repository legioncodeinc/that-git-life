# API contract conventions

- **Category:** Standards
- **Status:** Canonical
- **Last updated:** 2026-05-23

Defines the shape every REST endpoint under `http://localhost:3050/api/*` must follow. The full endpoint inventory lives in [`../service/api-contract.md`](../service/api-contract.md); this file is the rules-of-the-road.

---

## 1. Versioning

The current API version is **v1**. Every endpoint lives under `/api/v1/`. We bump to `/api/v2/` when a breaking change ships.

The web app only ever calls `/api/v1/*`. Internal CLI commands hit the same routes.

---

## 2. Request shape

- `Content-Type: application/json` for bodies. Bodies are JSON objects, never bare arrays at the top level.
- Query params for filtering/pagination, body for state changes.
- Pagination uses `?limit=<n>&cursor=<opaque>`. Default `limit=50`, max `limit=200`.

---

## 3. Success response

Every 2xx response is a JSON object with **exactly one** top-level key matching the resource:

```json
{ "repo": { "path": "/Users/m/GitHub/foo", "name": "foo", ... } }
```

For collections, the key is the plural noun and pagination metadata sits at the top level:

```json
{
  "repos": [ ... ],
  "nextCursor": "eyJ…" | null
}
```

Never return a bare array. Always wrap in an object so future fields can be added without breaking clients.

---

## 4. Error envelope

Every 4xx and 5xx response uses this shape:

```json
{
  "error": {
    "code": "REPO_NOT_FOUND",
    "message": "Human-readable explanation.",
    "details": { /* optional, structured */ }
  }
}
```

Error `code` values are `SCREAMING_SNAKE` and stable. The web UI matches on `code`, not on `message`. The catalog of codes lives in `src/shared/error-codes.ts`.

---

## 5. Status codes

| Code | When |
|---|---|
| 200 | Successful GET or successful action returning state. |
| 201 | Successful resource creation. Include `Location` header. |
| 204 | Successful action with no response body. |
| 400 | Malformed request (bad JSON, schema violation). |
| 401 | Missing or invalid local auth (see §7). |
| 403 | Authenticated but operation not permitted. |
| 404 | Resource not found. |
| 409 | Conflict (e.g., trying to add a repo already tracked). |
| 422 | Validation error on an otherwise well-formed request. |
| 500 | Unhandled server error (rare; logged at `error` level). |

---

## 6. Validation

Every route declares a Fastify JSON Schema for body, query, and params. Schema lives next to the route file. Schema violations return 400 with the field-level details in `error.details`.

---

## 7. Local auth

The service binds to `127.0.0.1` only (ADR-003). For defense in depth:

- All `/api/*` routes require a `X-TGL-Token` header.
- The token is generated at first service start and stored in `~/.tgl/api-token` (mode 0600).
- The web UI reads the token from a same-origin `/api/v1/handshake` endpoint that's only callable from `Origin: http://localhost:3050`.

This prevents drive-by attacks from a malicious page making fetch calls to `localhost:3050`.

---

## 8. Long-running operations

Operations that take more than ~5 s (e.g., scan-all-repos, sync-skills) return 202 with a job ID:

```json
{ "job": { "id": "job_01H…", "status": "running" } }
```

Clients poll `GET /api/v1/jobs/:id` for status. When the job completes, the response includes a `result` object.

We considered Server-Sent Events but kept polling for simplicity. Revisit if dashboard responsiveness suffers.

---

## 9. Time and IDs

- All timestamps are ISO-8601 strings in UTC (`2026-05-23T13:05:00Z`). Never Unix epoch in the API.
- IDs are ULIDs (`01H…`) generated by the server. The CLI doesn't invent IDs.
- Paths are absolute and use the OS's native separator. The web UI displays them as-is.

---

## 10. No magic

- The API never has implicit side effects beyond what its name implies. `GET` is read-only. `POST /scan` runs a scan.
- We don't return HTML from API routes. The web app is served at `/`; the API is `/api/*` and only returns JSON.
