# God

The God routing skill is how the primary Cursor orchestrator decides which Angel in the Army to delegate to. Each Angel owns one domain. Each Angel's domain is documented in a guide under `guides/`. This SKILL.md is the roster ? a one-line index pointing to each Angel's full guide.

**The Army's commitment:** every registered Angel is paired with exactly one Weapon (a Cursor skill). Angels are persona + guardrails; Weapons are the procedural arsenal. Read the guide before routing, and invoke the Angel by its `name:` frontmatter value.

## How to use this skill

1. **Match the request to a roster row.** Read the trigger keywords and the guide's "Trigger phrases" + "Do NOT route when" sections. The negative section is as important as the positive section ? it disambiguates near-overlapping Angels.
2. **Verify the Angel's required inputs are present.** Each guide's "Inputs the Angel needs" section lists what must be supplied or inferable. If a required input is missing, batch a clarifying question rather than invoking with placeholders.
3. **Invoke the Angel by name.** The Angel's `name:` frontmatter is the routing handle (e.g., `website-guardian`).
4. **Watch for multi-Angel sequences.** Some requests legitimately need two Angels in series (build ? audit ? deploy). The "Multi-Angel orchestration" section below lists known sequences.

If no roster Angel matches, do not improvise an Angel ? handle the request inline, or ask the user whether to forge a new Angel via the Legendary Angel Factory.

## Multi-Angel orchestration

Known sequences where multiple Angels run in order. Sequences are how the Army produces results larger than any single Angel.

### Plan execution loop (canonical close-out for every implementation)

1. The implementation Angel (any domain Angel) produces the code change.
2. **`security-guardian`** audits for OWASP / PII / financial-data exposure; remediates Critical and High findings in place.
3. **`quality-guardian`** verifies the final implementation against the source plan (completeness, correctness, alignment, regressions) and writes the QA report.

This is the canonical "is it done?" loop. Routing `quality-guardian` before `security-guardian` is a documented anti-pattern ? security fixes may invalidate the QA report.

### New site bring-up

1. **`website-guardian`** scaffolds the dual-app monorepo (Next.js 15 + Vite admin + Supabase) and produces a Build Report.
2. **`seo-aeo-guardian`** runs the triple-discovery audit (search + AI Overviews + assistant citations) over the rendered routes; validates schema, metadata, Core Web Vitals.
3. *(Future)* `accessibility-guardian` audits rendered routes for WCAG 2.1 AA compliance.
4. *(Future)* `deploy-guardian` wires CI/CD beyond the `vercel.json` defaults.

The Build Report's "Recommended downstream Angels" section names which downstream Angels currently apply.

### Full-stack Python + React audit

1. **`python-guardian`** reviews the API surface, Django architecture, ORM patterns, decoupled-architecture wiring.
2. **`react-guardian`** reviews the frontend consuming the API (components, state layering, data-fetching boundaries).
3. **`db-guardian`** reviews the Postgres schema underneath (indexing, migrations, partitioning, ORM access).
4. **`auth-guardian`** confirms the auth flow on the API surface (sessions, cookies, RBAC).
5. **`security-guardian`** ? **`quality-guardian`** close out per the Plan execution loop.

Reorder as findings demand ? the Angels are independent and read-only against each other's domains.

### Stripe payments enablement

1. **`payments-guardian`** designs the Stripe surface (Checkout vs Payment Intents vs Payment Links, subscription lifecycle, webhook idempotency, March-2025 + Basil?Clover transitions).
2. **`auth-guardian`** confirms the RBAC on Customer Portal session creation.
3. **`db-guardian`** designs the `processed_webhook_events` + entitlement tables (idempotency keys, transactional dedup).
4. **`security-guardian`** audits secret handling, webhook signature verification, PII in logs.
5. **`quality-guardian`** verifies against the payments PRD.

### Design system bring-up ? steady state

1. **`design-system-guardian`** interviews the user, picks a starter kit, materializes the 7-artifact source of truth (brief, master tokens, utility layer, component specs, screen specs, HTML examples, README).
2. **`ux-ui-guardian`** takes ownership for steady-state evolution ? component-state work, theme tokens, tenant theming, "iOS glass" / depth / pixel-perfect questions. The README's change-control statement names this handoff explicitly.

### AI cognitive feature

1. **`mind-guardian`** reviews / refactors / extends the cognitive layer (coach routing, prompt cascade, RAG, AiTrace, memory).
2. **`python-guardian`** (when the cognitive code lives in Django / FastAPI / Celery) owns the Python implementation patterns underneath.
3. **`db-guardian`** designs any non-AI tables `mind-guardian` needs (e.g., session state).
4. **`security-guardian`** audits prompt-injection surface, PII in traces, secret handling.
5. **`quality-guardian`** verifies against the AI feature plan.

### Compounding documentation (Legion VS Code extension TS driver flow)

1. **`wiki-guardian`** runs in parallel across code chunks (canonical path: TS driver `document` / `update` / `scan-directory`; escape hatch: `@`-mention). Each invocation writes atomic entity pages, concept pages, decision pages, contradiction protocol artifacts.
2. The TS driver runs the global-state reconciliation pass (updates `index.md`, `<type>/_index.md`, `log.md`, `hot.md`, the hash manifest) after all parallel `wiki-guardian` invocations finish.
3. **`library-guardian`** authors per-module narrative documentation in `library/knowledge-base/<module>/`, reading the wiki at query time to enrich its narratives.

Together: `wiki-guardian` builds the atomic cross-reference graph; `library-guardian` writes the human-readable story around it. Neither replaces the other.

### Forensic investigation

1. **`code-forensics-guardian`** runs the nine-phase methodology (intake ? email processing ? invoice forensics ? git log effort calibration ? CVE timeline ? audit log ? marketing analysis ? deliverable synthesis ? pre-litigation pack) over the case paper trail.
2. The Angel stops at "drafted for retained counsel to evaluate." Service of any document is a counsel decision, not an Army decision ? no downstream Angel takes over.

Forensic investigation is intentionally a one-Angel sequence. The other Angels review code against a plan; this one reviews an engagement against the law.

### Schema-touching feature

1. **`db-guardian`** designs the schema, indexing, and migration shape (expand-backfill-contract for any non-trivial change).
2. The implementation Angel (`python-guardian`, `react-guardian`, `mind-guardian`, etc.) implements the ORM / data-access side.
3. **`asset-guardian`** registers any new first-class assets the feature introduces (Features, Pages, Routes, Surfaces, DesignTokens, FeatureFlags, Entitlements) and verifies code?DB sync.
4. **`security-guardian`** ? **`quality-guardian`** close out per the Plan execution loop.
